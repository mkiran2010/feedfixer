# Data collection options

We've added a local verdict log (`chrome.storage.local`, capped at 1000 entries, exportable as JSON from the options page). Next question: should we ship this data anywhere, and if so where?

## What we currently log per scored reel

```ts
{
  videoId: "abc123XYZ_",
  title: "POV: when the rizz hits 😂😂",
  channel: "@somechannel",
  verdict: "Junk" | "Stay",
  level: 5,
  customRule: null | "<the user's custom instruction at score time>",
  scoredAt: 1715472000000
}
```

No personally-identifying data beyond what the user actively typed (their custom rule). YouTube videoIds are public.

## Three paths

### Path A — Local-only (current)

What we have now. The log lives in the user's browser. They can export it. We never see it.

| Pro | Con |
|---|---|
| Zero infra cost | Can't aggregate across users to learn what the population skips |
| Zero privacy concerns | Each user has to manually export to share data |
| Works offline | Lost if user uninstalls or wipes browser data |
| No backend to maintain | |

**Best for:** the current first-beta. Ship this, validate the product works, then decide if cross-user aggregation is worth the ops burden.

### Path B — Anonymous opt-in upload to S3

User toggles "Share my verdict log to help improve FeedFixer". Once a day (or on extension uninstall), the SW POSTs the log to a pre-signed S3 URL.

**What the user has to set up:**

1. AWS account
2. S3 bucket (private — we'll write to it via server-side credentials)
3. Either:
   - **Direct upload via pre-signed POST URLs** — generated by a tiny Lambda + API Gateway endpoint. The extension hits the endpoint to get a one-time signed URL, then PUTs the log directly to S3. ~30 lines of Lambda code, ~$0/month at hobby scale.
   - **Identity Pool via Cognito** — extension authenticates as anonymous identity, gets temp creds, uploads directly. More AWS surface area.
   - **Tiny custom backend** (Cloudflare Worker / Vercel Edge / Fly.io) — extension POSTs to your endpoint, your code writes to S3. Simplest mental model, ~50 lines.

**Recommended:** Cloudflare Worker writing to S3 via SDK. Free tier covers tens of thousands of requests/day. ~30 lines of TypeScript.

**Privacy considerations:**

- Need to add an opt-in toggle (default OFF) in the options page.
- Need to add a privacy policy disclosing what's uploaded and where.
- Need to decide whether to include any user-identifying token (e.g. random per-install UUID for de-duping). If yes, must be disclosed.
- Should NOT include the user's IP (S3/CDN logs will, though — mitigate via Cloudflare Worker stripping it before forwarding).

### Path C — Postgres / Supabase / Firebase

Same as B but the data lands in a queryable store you can run analytics on directly. Good if you want to look at patterns interactively.

| Pro | Con |
|---|---|
| Query with SQL or dashboard tools | More infra than S3 dump |
| Easy to compute "% of all users who skip this video" | Cost scales with rows |
| Built-in dedup, indexing | Schema migrations as we evolve the log shape |

**Recommended for path C:** Supabase. Free tier is generous (500MB DB, 2GB bandwidth/month). Add a `verdicts` table, hit a single `INSERT` endpoint. Row-Level Security policies can enforce "extension can only insert, never read others' rows."

## My recommendation

**Stay on path A for the first beta** (already shipped). Until you have enough users that aggregate data would actually tell you something interesting (~50–100 users), the infra burden of B/C isn't justified.

When you're ready to move to B or C, the lift is small:

- **Add to settings:** `uploadEnabled: boolean` (default false), `installId: string` (random UUID, generated once, persisted).
- **Add to SW:** a periodic alarm (`chrome.alarms.create('upload', {periodInMinutes: 1440})`) that POSTs the log if `uploadEnabled`.
- **Add to options:** an opt-in toggle and a "what gets uploaded" disclosure.
- **Add to manifest:** `host_permissions` for the upload endpoint and `"alarms"` permission.

That's ~50 lines of extension code plus whichever backend you pick. ~1 day of work end-to-end.

## What would actually make this decision

Three signals to wait for:

1. **N installs** — until there's a population, aggregation is moot.
2. **A specific question you want to answer** — "what % of POV titles get skipped?" "do users at level 7+ skip everything?" "what's the median Stay rate per user?" Don't build collection without a question.
3. **Validated request from users** — does anyone want a "community filter" / shared rubric mode? If yes, that's the killer feature collection unlocks.

Until one of those lands, path A is correct.
